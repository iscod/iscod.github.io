# Memcached 与常见缓存问题

Memcached、Redis 等内存缓存都能缓解数据库压力，但使用不当会出现**穿透**、**击穿**、**雪崩**三类典型问题。下文按「现象 → 思路 → 做法」说明；涉及具体 API 时以 Redis 为例（Memcached 可用 `add` 等实现类似互斥思路，需按实际引擎选型）。

---

## 1. 缓存穿透

### 现象

大量请求查询**客观上不存在**的数据（非法 ID、恶意构造的 key 等）。缓存不会命中，流量直接打到数据库，可能把库打挂。

### 思路

在到达缓存或数据库之前，尽早识别「不可能存在」的查询并短路返回。

### 做法

- **布隆过滤器**：在缓存前加一层布隆过滤器（例如用 Redis 的 Bitmap 等实现），对明显不存在的 key 直接返回，减少对数据库的无效访问，起到限流与拦截作用。
- **业务校验**：对明显非法参数在应用层直接拒绝，不进入缓存与数据库。

---

## 2. 缓存击穿

### 现象

某个**热点 key**（如热搜、秒杀商品）在过期瞬间，出现极高并发；缓存一旦失效，大量请求同时穿透到数据库。

**要点**：单点热点 key + 过期时刻 + 高并发。

### 思路

避免「同一时刻、大量线程同时重建同一份缓存」。

### 做法

- **热点逻辑上不过期**：物理 TTL 很长，或由后台任务定期刷新，避免在峰值时刻集体失效。（需配合监控与回源策略。）
- **互斥重建**：缓存失效时，只允许一个请求回源查库并回写缓存，其余等待或稍后重试。Redis 可用 `SET key value NX EX ttl` 等实现分布式锁。**优点**是对数据库冲击小；**缺点**是等待与重试会带来延迟。
- **逻辑过期**：在 value 里带「逻辑过期时间」；发现逻辑过期时仍先返回旧数据，同时异步更新缓存，用短期旧数据换平滑负载。

---

## 3. 缓存雪崩

### 现象

两类常见诱因：

1. **缓存层故障或大面积不可用**：实例宕机、网络分区等，流量全部涌向数据库。
2. **大量 key 在同一时刻过期**：TTL 设置过于整齐，到期瞬间集体失效，数据库压力陡增。

二者都可能引发连锁故障，即所谓「雪崩」。

### 思路

**高可用**降低缓存整体挂掉的概率；**打散过期**与**多级/限流**降低瞬时回源峰值。

### 做法

**针对缓存不可用**

- 使用**主从、集群**（如 Redis Sentinel / Cluster）做高可用，避免单点。
- **多级缓存**：本地缓存（如 Ehcache）+ 远程缓存，远端故障时仍有一定缓冲。
- **限流、熔断、降级**：接入层或网关限制并发，必要时拒绝部分请求，保护数据库与核心链路。

**针对集体过期**

- **随机抖动（Jitter）**：在基础 TTL 上增加随机区间，例如「1 小时 + 1～10 分钟随机」，避免同一时刻大面积失效。
- **双层缓存**：例如 A 层短 TTL、B 层长 TTL 作备份，降低同时全部失效的概率。
